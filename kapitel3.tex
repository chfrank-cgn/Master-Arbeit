\subsection{Kubernetes Security Signals}

Research defines security signals in IT Security as real-time indicators or alerts that highlight potential security threats, vulnerabilities, or anomalous behaviors within Kubernetes environments. Interpreting these signals is critical for detecting, responding to, and mitigating security risks across all infrastructure.\footnote {See \textit{Calloway, A. (2026)}: Kubernetes Security Signals. \cite{secSignals}}

In Kubernetes, security signals can be events, logs, or alerts generated by the Kubernetes cluster itself, by containers, or by applications that indicate unauthorized access attempts, misconfigurations, runtime threats, anomalous behavior, or vulnerabilities.

It is critical to proactively monitor these signals to maintain the security posture of a Kubernetes environment.

Security signals are most commonly generated using a combination of open-source and commercial tools, such as Falco, Aqua Security, and NeuVector for runtime security, and Calico, Cilium, and Istio for network security. 

Security signals from Kubernetes are then often forwarded to a SIEM (Security Information and Event Management) or SOAR (Security Orchestration, Automation, and Response) platform for correlation, automated responses, and compliance reporting.

For our experiment, we will focus on security signals from SUSE Security (formerly NeuVector; see \ref{neuvector}) by using the forwarding mechanism typically used in a classic SIEM platform.
