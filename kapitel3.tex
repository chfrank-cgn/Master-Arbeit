\subsection{Kubernetes Security Signals}

Research defines security signals in IT Security as real-time indicators or alerts that highlight potential security threats, vulnerabilities, or anomalous behaviors within Kubernetes environments. Interpreting these signals is critical for detecting, responding to, and mitigating security risks across all infrastructure.\footnote {See \textit{Calloway, A. (2026)}: Kubernetes Security Signals. \cite{secSignals}}

In Kubernetes, security signals can be events, logs, or alerts generated by the Kubernetes cluster itself, by containers, or by applications that indicate unauthorized access attempts, misconfigurations, runtime threats, anomalous behavior, or vulnerabilities.

It is critical to proactively monitor these signals to maintain the security posture of a Kubernetes environment.

Security signals are most commonly generated using a combination of open-source and commercial tools, such as Falco, Aqua Security, and NeuVector for runtime security, and Calico, Cilium, and Istio for network security. 

Security signals from Kubernetes are then often forwarded to a SIEM (Security Information and Event Management) or SOAR (Security Orchestration, Automation, and Response) platform for correlation, automated responses, and compliance reporting.

For our experiment, we will focus on security signals from SUSE Security (formerly NeuVector; see \ref{neuvector}) by using the forwarding mechanism typically used in a classic SIEM platform.

\subsection{Syslog}

Syslog-ng and rsyslog are the two most popular syslog server implementations for Linux environments, but they differ in key ways. 

Syslog-ng is more feature-rich and flexible, with a powerful configuration language, built-in support for message parsing and reformatting, and the ability to store log messages in various databases. It also has a commercial version with additional features and support. 

Rsyslog is lighter and faster, designed to handle large volumes of log messages efficiently. It has a simpler configuration language and relies on external plugins for advanced functionality. Rsyslog is fully open-source and free to use.

Both would have been equally good choices; in the end, the available destinations made syslog-ng the better fit.\footnote{See \textit{syslog-ng OSE (2026)}: syslog-ng destination options. \cite{ngDestinations}}

In this setup, Syslog-ng runs a single-replica deployment and exposes the standard syslog TCP port 514 as a service, as configured in NeuVector above.

\begin{lstlisting}[caption=Syslog Service Definition, frame=single, basicstyle=\ttfamily\small, breaklines=true]
apiVersion: v1
kind: Service
metadata:
  name: syslog
  namespace: qradar
spec:
  ports:
    - port: 514
      targetPort: 514
      protocol: TCP
  selector:
    app: syslog

\end{lstlisting}

\subsection{MQTT}

Given the high volume of data, it's good practice and an established design pattern to use a message broker.\footnote {See \textit{Calloway, A. (2026)}: When to use a Message Broker. \cite{mqttPattern}}

Eclipse Mosquitto is the dominant, most widely used open-source, lightweight MQTT broker. MQTT brokers use a publish-subscribe (pub/sub) messaging model. A service publishes a message to the broker, which then routes it to the appropriate subscribers.\footnote {See \textit{mosquitto.org (2026)}: What is an MQTT Broker. \cite{whatBroker}}

In this setup, Mosquitto runs as a single-replica deployment, exposing the standard MQTT port 1883 via a cluster service.

To receive the event stream, we instruct syslog-ng to publish each individual signal to the topic qradar/neuvector of our broker:

\begin{lstlisting}[caption=Destination d-mqtt, frame=single, basicstyle=\ttfamily\small, breaklines=true]
destination d_mqtt {
    mqtt(
    address("tcp://mosquitto:1883"),
    topic("qradar/neuvector"),
    fallback-topic("syslog/fallback"),
    template("$$MESSAGE"));
};

\end{lstlisting}

\subsection{Ollama}

As of this writing, several tools are available for running AI models locally.\footnote {See \textit{Calloway, A. (2026)}: Common Applications to Run AI Models Locally. \cite{localLlama}}

There is Ollama, an open-source tool to run LLMs locally. Ollama supports models such as Gemma, Llama, Mistral, and Qwen and provides a widely publicized API for programmatic access.

Then there is LM Studio, a GUI-based tool for downloading and running LLMs locally. There is also llama.cpp, a C++ tool and library for running LLMs efficiently locally on CPUs/GPUs, and finally vLLM and KoboldCpp.

Ollama is part of the SUSE AI stack and provides ready-made Kubernetes deployment files, making it an easy choice.\footnote {See \textit{SUSE (2026)}: SUSE AI Stack. \cite{suseAI}} Self-hosting, though, is not for everyone, even on Kubernetes.\footnote{See \textit{Davies, N. (2026)}: Self-Hosted LLMs in the Real World. \cite{selfHost}}

Several factors should be considered when selecting a local model, which we will cover in the following chapters.\footnote{See \textit{Conway, A. (2026)}: A local LLM for coding. \cite{localModel}} For the setup, we will start with Gemma3 as our model.

Ollama is currently embroiled in controversy over license compliance, attribution, and transparency regarding its use of the open-source project llama.cpp. Several voices argue that one should not continue to use Ollama, especially if one prioritizes transparency and performance.\footnote{See \textit{Zetaphor (2025)}: Friends don't let Friends use Ollama. \cite{ollamaDrama}}

In the light of this controversy, one could consider running llama.cpp directly instead, or explore other tools such as vLLM or KoboldCpp.

\subsection{Bridge}

With the message broker and the API in place, we now need to integrate them. For this purpose, we will use a Python program that subscribes to the neuvector/qradar topic on our MQTT broker and forwards the message to Ollama.

\begin{lstlisting}[caption=Event Bridge Excerpt, frame=single, basicstyle=\ttfamily\small, breaklines=true, language=Python]
from openai import OpenAI
import paho.mqtt.client as paho

# Connect callback
def on_connect(mosq, obj, flags, reason_code, properties=None):
  client.subscribe("qradar/neuvector")

# Ollama connect
ollama = OpenAI(base_url='http://ollama.ollama.svc:11434', api_key='ollama')
model = 'gemma3'
messages = [ 
  {"role": "system", "content": "You are a helpful Kubernetes security expert."}
]

\end{lstlisting}

Once Neuvector sends a signal to our SIEM via Syslog, it reaches our bridge via the MQTT broker, enabling us to act on it immediately.

\subsection{Elasticsearch}

Elasticsearch has long been the gold standard for log file storage and analysis in IT.\footnote{See \textit{Elastic (2026)}: Elastic recognized as a Leader. \cite{gartnerElastic}} While there are many alternatives, such as Grafana Loki or Splunk, many IT operations rely on Elasticsearch or its forked cousin, OpenSearch.

In our setup, we will use an Elasticsearch instance for long-term storage of security events and for forensic analysis.

Syslog-ng allows events to be sent directly to Elasticsearch, with no post-processing needed.

\begin{lstlisting}[caption=Destination d-elastic, frame=single, basicstyle=\ttfamily\small, breaklines=true]
destination d_elastic {
    elasticsearch-http(
        index("neuvector"),
        op_type("create"),
        batch-bytes(0),
        batch-lines(0),
        url("http://elasticsearch:9200/_bulk"));
};

\end{lstlisting}

The standard interface for Elasticsearch data is Kibana, and without defining any dashboards, we can analyze security events in Discover.

\subsection{MCP Server}\label{mcp-view}

As a companion to our LLM, we can run an MCP Server. A Kubernetes MCP server can significantly enhance security event analysis by providing the LLM with access to all cluster metadata, including information about nodes, pods, and containers. This metadata can be very useful for enriching security log data and making it easier to analyze. However, as we move from text logic to container orchestration, we must ensure that appropriate security safeguards are in place. From Domain-Driver Design, we can borrow Bounded Contexts and the Anti-Corruption Layer.\footnote{See \textit{Traub, D. (2026)}: Rediscovering Domain-Driven Design. \cite{domainDriven}}

\begin{figure}[H]
\centering
\caption {MCP Server}
\includegraphics[width=0.9\linewidth]{images/mcp-server.png}
\label{fig:mcp-server}
\end{figure}

In our setup, to aid the LLM in interpreting Neuvector signals, we added an open-source Kubernetes MCP server from the Containers repository restricted to read-only access, using Kubernetes' Role-Based Access Control.\footnote{See \textit{Containers (2026)}: Welcome to Containers. \cite{whichMCP}} RBAC is a security mechanism that regulates access to resources based on the roles assigned to users, groups, or service accounts.\footnote{\textit{Ferraiolo, D. et.al. (2007)}: Role-Based Access Control. \cite{rbac}}

RBAC in Kubernetes allows administrators to define granular permissions by creating Roles and ClusterRoles that specify which actions (e.g., get, create, delete) are allowed on which resources (e.g., pods, deployments, secrets). These roles are then bound to users or service accounts via RoleBindings (namespace-scoped) or ClusterRoleBindings (cluster-wide):

\begin{lstlisting}[caption=Cluster View Role Definition, frame=single, basicstyle=\ttfamily\small, breaklines=true]
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  labels:
    app.kubernetes.io/name: kubernetes-mcp-server
    app.kubernetes.io/instance: mcp-server
  name: mcp-server
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: view
subjects:
- kind: ServiceAccount
  name: mcp-server
  namespace: mcp-server
  
\end{lstlisting}

\subsection{Air Gap}

To support regulated industries, the entire setup can be deployed in an air-gapped environment without connection to the public internet.

However, there are some caveats: After its initial training, the model's knowledge quickly becomes outdated, which might affect the accuracy of its decision-making.

As discussed in \ref{threat-intelligence}, it's important to augment the model's knowledge with current information, especially when the model itself cannot access an up-to-date CVE database or a list of current threat vectors.\footnote{See \textit{Bowie, C. et al. (2026)}: Inside the Human-AI Feedback Loop. \cite{feedbackLoop}}

Another factor to consider for long-term usage is model bias.\footnote{See \textit{Xu, J. et al. (2026)}: AI self preference. \cite{xu2026}} If an offline model needs to be updated or replaced, it might disfavor previous decisions or analyses made by a different model.

\subsection{GDPR}

The setup itself does not generate any personally identifiable information and therefore should not fall under the GDPR. 

In production, a detailed inspection of the event-stream data will be necessary to identify any PII collected from the application that may appear in the security events sent, and the system may need to be reclassified.
