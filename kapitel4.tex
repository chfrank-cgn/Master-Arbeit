\section{Static AI}

\onehalfspacing

\subsection{State of the art}

It's a general sentiment in the industry that AI models are increasingly taking over specific cybersecurity tasks, not to replace human experts entirely but to handle repetitive, data-heavy, or time-sensitive work more efficiently. While AI excels at tasks such as log analysis, anomaly detection, and threat triage—where it can sift through vast datasets to identify patterns and flag suspicious activity—it still struggles with contextual judgment, strategic decision-making, and understanding nuanced attack vectors that require human intuition.\footnote{See \textit{Claburn, T. (2026)}: AI models replacing cybersecurity pros. \cite{aiSecurity}}

AI can be particularly useful for automating mundane tasks, such as filtering false positives in alerts or patching known vulnerabilities, thereby freeing up time for more complex work, such as threat intelligence, incident response planning, or red teaming.

As of the time of writing, AI's role is more that of augmentation rather than replacement for cybersecurity teams. The most effective approach combines AI's speed and scalability with human expertise, ensuring that critical decisions, such as responding to a zero-day exploit, remain in human hands. For now, AI seems to be more of a force multiplier than a standalone option. We will explore this role in this chapter.

In the next chapter, we will examine options to give the AI model greater leeway in decision-making.

\subsection{Context Window}

The main issue with context and memory in large language models is that they often struggle to maintain coherence and consistency over longer conversations or documents. LLMs are trained on large amounts of text data, but each piece of text is treated independently.

So while an LLM can generate fluent and relevant responses to short prompts, it doesn't inherently retain memory of the full conversation context. This means that, as a conversation progresses, an LLM may lose track of key earlier details or even contradict itself. In our case, that would mean the LLM would lose the history of past events over time.

An LLM can quickly interpret a single security signal; however, it may not know whether the same signal occurred last week. This is why the setup includes a data lake for long-term viewing and reporting.

Researchers are working to improve LLMs' ability to handle long-term context and memory. \footnote{See \textit{Himansha, R. (2026)}: Claude's Long Context Window. \cite{claudeContext}} Some approaches include providing the model with a memory buffer that stores conversation history, fine-tuning the model on coherent long-form text to improve consistency, and experimenting with novel neural architectures that better track context. More memory alone will most likely not be sufficient to fully capture the context of a single security event.\footnote{See \textit{Martinez, S. (2026)}: More Memory Won’t Fix Your AI Agents. \cite{moreMemory}}

\subsection{Model Selection}

Earlier in this paper, we stated that we would start with the Gemma3 family of LLMs. Gemma3 is designed to be smaller and faster than models such as Llama 3 or Mistral, making it well-suited for resource-constrained environments, e.g., air-gapped systems in regulated environments. This efficiency is particularly useful for real-time log analysis and automated threat detection with low latency.\footnote{See \textit{Calloway, A. (2026)}: Gemma3 for IT Security. \cite{gemma3}}

They also state that, with strong performance on structured tasks, Gemma3 will excel on text-based tasks. Gemma3 can parse and correlate logs to identify anomalies or indicators of compromise (IoCs) and create concise summaries of security events for SOC teams. If it's not completely air-gapped, it can extract key insights from threat reports or vulnerability databases.

Gemma3 can thus also generate concise summaries of security events, which will be a key element of our setup.

They also found that Gemma3's context window is smaller than that of models such as Llama 3 and Claude 3. This will affect its ability to analyze complex attack chains across multiple logs or over longer periods, but for now, we will stick with the Gemma3 family.

To select a model from the Gemma3 family, we must consider the available hardware. For this paper, we use a Standard\_NC6s\_v3 system as the node on which Ollama will run. This system will provide an NVIDIA Tesla V100 GPU with 16GB of VRAM.\footnote{See \textit{Azure (2026)}: NCv3 size series. \cite{ncv3Series}}

The Gemma3 family of models in the Ollama library comprises variants with different quantizations and memory requirements.\footnote{See \textit{Ollama (2026)}: Gemma3. \cite{modelsGemma3}}

The gemma3:12b model with Q4\_K\_M quantization seems to be the largest member of the Gemma 3 family that will run fine on our system with the NVIDIA V100 GPU.\footnote{See \textit{Will It Run AI (2026)}: Gemma 3 12B. \cite{nvidiaV100}}

Let's look at the model designation in detail:

\begin{itemize}
    \item 12B refers to the number of parameters in the model, in this case, 12 billion parameters 
    \item Q stands for quantization, and 4 indicates 4-bit precision
    \item K identifies the quantization method, k-means clustering or k-block quantization
    \item M stands for "memory-efficient" or "mixed" quantization
\end{itemize}

Gemma3:12B with Q4\_K\_M quantization appears suitable as a lightweight LLM for security log analysis in clusters and is compatible with our hardware setup. It will, however, not perform at the level of Anthropic's Mythos.\footnote{See \textit{Ahmad, M. (2026)}: Mythos AI is a cybersecurity threat. \cite{mythosThreat}}

With sufficient time and resources, one could train a model oneself; however, this process is outside the scope of this paper.

\subsection{Incident Handling}

Our security signals are available in the message broker queue and the long-term data lake.

Let's start analyzing an event. First, we will provide the LLM an introductory system prompt through our bridge:\footnote{See \textit{Khalesi, B. (2026)}: I finally left ChatGPT for Gemini. \cite{toGemini}}

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
You are a helpful Kubernetes security expert.
        
Reply concisely.

Don't over-explain basic answers.
        
Use direct language.
        
Preserve technical formatting.
    }
}

Then we pass the instructions for each event, followed by the event itself.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
Here is a security event from NeuVector. Please analyze.
     }
}

The LLM will then respond to our bridge with a detailed analysis of the event in the customary markdown format.

There are many libraries available for Python to read markdown and extract the formatting, such as the Markdown Worker.\footnote{See \textit{Khurana, M. (2025)}: Markdown Worker. \cite{markdownWorker}}

First, we will get a summary of an event that we sent to the LLM. In the prompt, we instructed it to be concise and keep technical formatting.

\indent\colorbox{codegrey}{

\#\# NeuVector Security Audit Analysis
    }
}

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Summary

Severity: Critical (driven by CVSSv3 9.8)

Type: Container Image Vulnerability
     }
}

The event contains an image vulnerability classified as critical in the Common Vulnerability Scoring System, with a score of 9.8 on a scale from 0 to 10.

Next, we get the affected workload in an easy-to-read format.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Affected Workload

\textbar{} Field \textbar{} Value \textbar{}

\textbar{}-------\textbar{}-------\textbar{}

\textbar{} Namespace \textbar{} cattle-gatekeeper-system \textbar{}

\textbar{} Image \textbar{} openpolicyagent-gatekeeper:v3.13.0 \textbar{}

\textbar{} Base OS \textbar{} debian:11 \textbar{}
     }
}

After the information about the affected workload, we will receive detailed information about the observed vulnerability.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Vulnerability Detail

\textbar{} Field \textbar{} Value \textbar{}

\textbar{}-------\textbar{}-------\textbar{}

\textbar{} CVE \textbar{} CVE-2024-24790 \textbar{}

\textbar{} Package \textbar{} go:stdlib @ 1.20.7 \textbar{}

\textbar{} CVSSv3 Score \textbar{} 9.8 (Critical) \textbar{}

\textbar{} Vector \textbar{} AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H \textbar{}

\textbar{} Fix Available \textbar{} $\ge$ 1.21.11 or $\ge$ 1.22.4 \textbar{}

Root Cause: Go stdlib net/netip IP classification methods (IsPrivate, IsLoopback, etc.) mishandle IPv4-mapped IPv6 addresses (e.g.,::ffff:192.168.1.1), returning incorrect results compared to their IPv4 equivalents.
    }
}

Based on this information, our model performs a risk assessment.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Risk Assessment

This is OPA Gatekeeper - a policy enforcement component. Bypassing its IP-based logic could undermine admission control rules that rely on network classification.

- Attack Vector: Network (no auth required, no user interaction)

- Impact: Full C/I/A compromise if exploited in context

- Exploitability: Low complexity - attacker sends IPv4-mapped IPv6 addresses to bypass IP-based policy checks
    }
}

Based on that risk assessment, the model will recommend remediation.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Remediation

Immediate: Upgrade to a Gatekeeper version built with Go $\ge$ 1.21.11 or $\ge$ 1.22.4

Check the current version and upgrade openpolicyagent-gatekeeper to a release that bundles a patched Go stdlib. 

OPA Gatekeeper v3.14.0 or newer is recommended.
    }
}

Based on the risk assessment and remediation, our model assigns the event a high priority and recommends immediate patching.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Priority

High Priority - Network-exploitable, no privileges required, affects a security-critical component (policy enforcement). Patch immediately.
    }
}

\subsection{Using LLM for triage}

Triage is a concept from the medical field that involves prioritizing patients based on the urgency of their care needs.

In the context of IT security, triage works similarly, but with security incidents and vulnerabilities instead of medical needs. It's the process of assessing and prioritizing security issues based on their severity and potential impact.

To let the model perform triage and identify which security events we need to act upon and which events we can ignore, we can add a line to our message and ask the model after sending the event details.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
Should I take any action?
     }
}

For the first event, our model recommends taking action.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Bottom Line

Yes, take action - but it's not urgent or an emergency. Track the Rancher release for an updated Gatekeeper image and schedule an upgrade. Monitor in the meantime.
     }
}

For the second event, no action is recommended.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Bottom Line

No patching needed.
    }
}

So from our examples, we have one security event with medium priority and one that we can safely ignore.

\subsection{Using LLM for creating internal incidents}

Having the model decide whether to create an internal incident for a security event will help reduce alarm fatigue if it can classify events correctly. Kubernetes and SUSE Security have been around for some time and should be in the training data of most, if not all, currently available models.

However, accurately assessing the severity of a CVE requires up-to-date information, which may conflict with the need for an air-gapped environment in regulated industries.

SUSE Security itself has a mechanism to update its CVE database by reloading one of its images, which would need to be transferred manually every day in a fully air-gapped environment.\footnote{See \textit{SUSE (2026)}: Update the CVE Database. \cite{scannerUpdate}} The process used for this update could also be extended to perform regular model updates.

\begin{figure}[H]
\centering
\caption {CVE Database}
\includegraphics[width=0.9\linewidth]{images/cve-database.png}
\label{fig:cve-database}
\end{figure}

From the model, we obtain the priority of the recommended remediation, which we then can augment with data from the MCP server. On a priority scale of None, Low, Medium, High, and Critical, we could introduce a cutoff at Medium if the workload is still active, and at High if it is no longer active.

Based on this decision, the bridge could be enhanced to use the incident analysis from our model and create an incident by calling the ITSM tool's REST API using the Python Requests package.\footnote{See \textit{BMC (2026)}: Using the platform REST API to create an incident entry. \cite{bmcCreate}}\textsuperscript{,}\footnote{See \textit{Python Software Foundation (2026)}: Requests. \cite{pythonRequests}}

Events below the cutoff would not create an incident; however, they would still be available in the long-term data storage for forensic analysis and summary reports.

In our examples above, for the first event, it would mean creating an internal incident, as the event had medium priority while the workload was still active. For the second, low-priority event, no event would be created.

\subsection{Reporting}\label{breach}

So far, we have only handled internal events. We now need to add another triage for actual breaches, to create the formal incident reports that NIS2 requires.

If an incident is critical and affects the availability of critical services, it will need to be reported to the BSI, using a NIS2-Compliant Incident Response Template.\footnote{See \textit{Calloway, A. (2026)}: Incident Response Plan Template. \cite{responsePlan}} While we will develop a bespoke approach in this paper, it's worth noting that several commercial solutions already offer structured workflows and audit-proof records for this.\footnote{See \textit{Formalize (2026)}: Von IT‑Grundschutz zu NIS-2. \cite{formalize}}

Now, let's look at a critical event.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\# NeuVector Security Incident Analysis
    }
}

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Summary

Severity: Critical

A privileged container named nsenter-ipp9hg spawned an interactive bash shell via nsenter, violating its process profile.     
     }
}

Spawning a shell and gaining access to the host's operating system is indeed a very serious breach.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Vulnerability Detail

\textbar{} Indicator \textbar{} Risk \textbar{}

\textbar{}-----------\textbar{}------\textbar{}

\textbar{} nsenter as parent process \textbar{} Host namespace escape tool \textbar{}

\textbar{} bash -l (login shell) \textbar{} Likely human operator or backdoor \textbar{}

\textbar{} proc\_effective\_user=root \textbar{} Full root privileges \textbar{}

\textbar{} alpine:latest image \textbar{} Generic base image \textbar{}

\textbar{} workload\_domain=default \textbar{} Running in default namespace \textbar{}

\textbar{} Pod name pattern nsenter-ipp9hg \textbar{} Typical of manual pod injection \textbar{}
    }
}

Our model identifies two possible attack patterns.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Attack Pattern

Legitimate but dangerous: kubectl debug or manual ephemeral pod used for node debugging
   
Malicious: Attacker with kubectl exec and pod create permissions attempting host namespace breakout
    }
}

It also proposes immediate actions.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Immediate Actions

Identify who created this pod

Check audit logs for pod creation

Kill the pod immediately if unauthorized

Check for host-level impact on the node
    }
}

And offers its final verdict.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\#\# Verdict

High-confidence host namespace escape attempt or unauthorized privileged debugging session. Treat it as a compromise until proven otherwise.
    }
}

For this paper, we will assume that a malicious actor gained access to the host operating system and exfiltrated customer data.

\subsection{Using LLM for reporting}\label{reporting}

As our model identified this as a serious breach and the human IT Security staff supported this judgment, it's now time to report the incident to the BSI. As outlined in \ref{reporting-reqs}, there are three reports to be generated:

\begin{itemize}
    \item Early Warning
    \item Incident Notification
    \item Final Report
\end{itemize}

According to Article 23 of the NIS2 directive, the Early Warning must be submitted without undue delay and within 24 hours of becoming aware of a significant incident.

The Incident Notification (Detailed Report) must then be submitted within 72 hours of the early warning.

And the Final Report (Post-Incident Review) must be submitted within one month of the incident notification (or longer if justified by complexity).

Let's begin with the Early Warning, using the templates mentioned earlier.\footnote{See \textit{Calloway, A. (2026)}: Incident Response Plan Template. \cite{responsePlan}} We have made the templates available to our model.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
From the previous event, create a report using the NIS2 Early Warning Template.
     }
}

The model will respond by filling in the fields from the incident.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\# Early Warning

Subject: Early Warning - Cybersecurity Incident (NIS2)

Incident Time: 2026-05-25T15:46:34Z

Severity: Critical

Nature of Incident: Host namespace breakout

Initial Impact: Data breach

Contact: IRT Lead
    }
}

Depending on the setup and processes, the model could either send the Early Warning to the BSI itself or wait for human interaction.

At the latest, after 72 hours, we need to provide a detailed incident report.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
From the previous event, create a report using the NIS2 Incident Notification Template.
     }
}

The model will respond by filling in the incident fields, and the IT security team will need to add details about the presumed data breach.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\# Incident Notification

Subject: Incident Notification

Description: nsenter is specifically used to enter host namespaces (PID, network, mount, etc.). Executing bash -l as root after nsenter execution strongly suggests active host-level compromise or reconnaissance.

Impact: Host Operating System access on the main customer database server, suspect exfiltration of customer payment details

Mitigation Measures: The workload was identified and deleted

Next Steps: Review the affected node
    }
}

The Incident Report needs to be sent to the BSI, and within a month, we need to submit the final report.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
From the previous event, prepare a final report using the NIS2 Final Report Template.
     }
}

The model will prepare the final report, which then needs to be augmented by the IT security team, as the model receives only Kubernetes security signals and has access only to the Kubernetes layer.

\indent\colorbox{codegrey}{
    \parbox{0.8\linewidth}{
\#\# Final Report

Subject: Final Incident Report

Root Cause Analysis: An unprotected namespace allowed a malicious host namespace breakout

Impact Assessment: Several Megabytes of customer payment data were exfiltrated before the breach was closed

Response Timeline: Within a minute of the incident, the malicious pod was identified and deleted. The cluster was quarantined, and the database was checked. Affected customers were notified.

Lessons Learned: 

Enforce PodSecurity restricted or baseline in the default namespace.

Restrict hostPID: true via OPA Gatekeeper.

Audit RBAC and identify who has pods/create in the default namespace

Evidence Preserved: All security events from the past two hours leading up to the incident were exported from Elasticsearch; also, all container logs from the same time frame were exported
     }
}

For the final report, it's crucial not only to process the security signals as they occur but also to retain them for long-term analysis. The event data and model responses in our bridge are ephemeral and might age out soon.

\subsection{Summary}

Our experiment has shown that a local LLM can support the IT security team in handling security incidents by evaluating the security signals sent from SUSE Security. 

It can also support the team by preparing the necessary information for NIS2 reports through automated data collection and templating.
